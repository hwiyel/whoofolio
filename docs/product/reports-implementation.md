# 보고서 구현 메모

## 목적

이 문서는 `reports.md`의 기능 범위를 실제 구현 관점으로 풀어쓴 메모다.  
기획 문서가 "무엇을 만들지"를 다룬다면, 이 문서는 "지금 무엇이 구현되어 있고 왜 그렇게 만들었는지"를 기록한다.

## 빠른 요약

- 월별/일별 추이 계산은 `report_summary.json`
- 연도별 상위 수입원/지출원 계산은 `report.json`
- 분석 화면 기준 기간은 `재작년 / 작년 / 올해`
- 추이 원본 데이터는 최근 `36개월`
- 상위 수입원/지출원은 그룹을 제외한 실제 계정만 사용
- 월간분석 cold path는 최적화 후에도 월 아이템 상세 때문에 추가 비용이 남아 있다

## 현재 구현 범위

현재 분석 보고서는 아래 3개 화면을 중심으로 동작한다.

- `월간분석`
  - 이번 달 수입, 지출, 순현금흐름
  - 전월 비교
  - 월별 수입/지출 추이
  - 일별 현금흐름
  - 월 아이템 목록
- `수입분석`
  - `재작년 / 작년 / 올해` 기준 월별 수입 막대 + 누적선
  - 상단 요약 카드 `수입총계 / 수입평균 / 수입원`
  - 우측 `Top 5 수입원`
- `지출분석`
  - `재작년 / 작년 / 올해` 기준 월별 지출 막대 + 누적선
  - 상단 요약 카드 `지출총계 / 지출평균 / 지출원`
  - 우측 `Top 5 지출원`

## 코드 기준 진입점

현재 보고서 기능을 볼 때 가장 먼저 확인할 파일은 아래와 같다.

- 백엔드 응답 조립
  - `app/api/internal/reports/service.go`
- 후잉 API 클라이언트
  - `app/api/internal/whooing/client.go`
  - `app/api/internal/whooing/types.go`
- 프론트 보고서 화면
  - `app/web/src/pages/reports-page.tsx`
- 프론트 API 타입
  - `app/web/src/lib/api.ts`

## 후잉 API 사용 원칙

보고서 구현에서는 `report_summary.json`과 `report.json`을 역할별로 나눠 쓴다.

### 1. 추이와 합계는 `report_summary.json`

아래 데이터는 계정별 분해보다 숫자 집계가 중요하다.

- 월별 수입 추이
- 월별 지출 추이
- 일별 현금흐름
- 현재월/전월 요약

이 구간은 `report_summary.json`의 `rows_type=month|day`를 사용한다.

- 이유
  - 응답이 더 작다
  - 파싱이 단순하다
  - 반복 호출을 줄이기 쉽다

주요 사용 대상:

- `monthlyIncomeTrend`
- `monthlyExpenseTrend`
- `dailyCashflowTrend`
- 현재월/전월 요약 계산용 월 합계

### 2. 상위 수입원/지출원은 `report.json`

아래 데이터는 계정별 합산 금액이 필요하다.

- 연도별 상위 수입원
- 연도별 상위 지출원
- 수입원/지출원 집중도

이 구간은 `report.json`의 `rows_type=year`를 사용한다.

- 이유
  - `results.rows[연도].income.accounts`
  - `results.rows[연도].expenses.accounts`
  - 같은 구조로 연도별 계정 합계를 바로 읽을 수 있다

주요 사용 대상:

- `yearlyIncomeSources`
- `yearlyExpenseSources`

## 응답 가공 규칙

후잉 원본 응답을 프론트에 그대로 넘기지 않고, 보고서 화면에서 바로 쓰기 쉬운 구조로 한 번 가공한다.

### 월간 보고서 응답

월간 보고서 API는 기본 월간 응답 외에 아래 구조를 함께 내린다.

- `monthlyIncomeTrend`
- `monthlyExpenseTrend`
- `dailyCashflowTrend`
- `yearlyIncomeSources`
- `yearlyExpenseSources`

### 연도별 상위 항목 구조

`yearlyIncomeSources`, `yearlyExpenseSources`는 아래 형태를 기준으로 쓴다.

- 연도별 row 1개
- 대표 항목 1개
- 상위 리스트 최대 5개

프론트는 이 구조를 그대로 카드와 우측 패널에 쓴다.  
즉, 화면에서 다시 정렬하거나 계정 타입을 분리하지 않도록 백엔드에서 최대한 정리해 내려주는 쪽을 택했다.

## 계정 처리 규칙

`report.json`의 `accounts`에는 실제 계정과 그룹 계정이 함께 섞여 내려온다.

- 예: `주수입`, `부수입/기타` 같은 그룹
- 예: `월급`, `지원금`, `투자수익` 같은 실제 계정

보고서의 `Top 5 수입원`, `Top 5 지출원`은 그룹을 제외하고 실제 계정만 사용한다.

- 사용 기준
  - `accounts.json`으로 `account_id -> title/type` 매핑
  - `type=account`만 유지
  - 금액 내림차순 정렬
  - 상위 5개만 노출

이 규칙을 둔 이유는 사용자가 "어디서 들어오고 어디로 나가는지"를 바로 읽게 하기 위해서다.  
그룹까지 섞으면 `주수입`, `부수입/기타` 같은 중간 집계가 상위권에 끼어들어 실제 해석력이 떨어진다.

## 기간 기준

초기에는 `최근 1년` 중심으로 차트를 보려 했지만, 분석 카드의 전년 비교까지 같이 보여주려면 3개년 원본 데이터가 더 맞았다.

현재 기준은 아래와 같다.

- 프리셋: `재작년 / 작년 / 올해`
- 월별 원본 추이: 최근 `36개월`
- 연도별 구조 분석: 최근 `3개년`

이 구조로 바꾼 이유는 다음과 같다.

- `올해`는 `작년`과 비교해야 한다
- `작년`은 `재작년`과 비교해야 한다
- 차트와 카드 계산이 같은 시간축을 공유해야 한다

주의할 점은 `최근 1년`처럼 rolling window를 쓰지 않는다는 점이다.  
현재 분석 화면은 캘린더 연도 기준 비교를 전제로 설계되어 있다.

## 성능 최적화 메모

초기 월간분석의 큰 병목은 후잉 API를 월별/계정별로 반복 호출하던 부분이었다.

### 변경 전

- 24개월 추이를 월별 요약 API로 24회 호출
- 일별 현금흐름을 계정별로 반복 호출

### 변경 후

- 24개월 추이 대신 `36개월` 월별 집계를 `report_summary.json rows_type=month` 1회로 조회
- 일별 현금흐름을 `report_summary.json rows_type=day` 1회로 조회

개발 중 실측 기준으로 cold path는 대략 `8~9초`에서 `3~4초` 수준까지 줄었다.  
다만 월 아이템 목록과 카테고리 상세는 여전히 추가 비용이 있어, 첫 요청이 아주 빠른 상태는 아니다.

현재 남아 있는 주요 비용은 아래 쪽에 가깝다.

- 월 아이템 목록 상세 조회
- 카테고리 상세 조립
- 월간분석 첫 요청 시 캐시 미스 구간

## 현재 제약

- `수입원`, `지출원` 카드 표현은 아직 미확정이다
- 월 아이템 목록은 초기 응답에 포함되어 있어 cold path가 남아 있다
- 상위 수입원/지출원은 현재 연도 기준 요약이며, 월 단위 구조 분석까지 확장되진 않았다

## UI 메모

### 수입/지출 추이 차트

- 막대: 월별 금액
- 실선: 누적 금액
- 전년 비교 점선은 실험 후 제거

점선을 뺀 이유는 한 차트에 현재값, 누적값, 비교값이 동시에 들어가면 읽기성이 떨어졌기 때문이다.

### 상단 카드

- 수입분석: `수입총계 / 수입평균 / 수입원`
- 지출분석: `지출총계 / 지출평균 / 지출원`

`수입원`, `지출원` 카드는 현재 의도적으로 비워둔 상태다.  
집중도를 숫자, 도넛, 짧은 문장 중 어떤 형태로 둘지 아직 확정하지 않았기 때문이다.

### 우측 패널

- 수입분석: `Top 5 수입원`
- 지출분석: `Top 5 지출원`

이 패널은 차트 옆에서 "무엇이 큰가"를 바로 읽는 보조 영역으로 유지한다.

## 남은 과제

- `수입원 / 지출원` 카드 표현 방식 확정
- 월 아이템 목록을 계속 초기 응답에 포함할지 재검토
- 카테고리 상세를 지연 로딩으로 분리할지 판단
- 월간분석 cold path를 더 줄이기 위한 캐시 전략 세분화

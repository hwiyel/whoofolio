# 기술 스택 결정

## 최종 권장안

초기 구현 기준 권장 스택은 아래와 같다.

- 프론트엔드: React + TypeScript + Vite
- 데이터 조회/캐시: TanStack Query
- 테이블 UI: TanStack Table
- 스타일: Tailwind CSS
- 차트: Recharts
- 백엔드: Go + Chi
- 데이터베이스: Postgres
- 프록시: 기존 Nginx Proxy Manager
- 배포: Docker Compose

## 각 기술의 역할

### React

브라우저 UI를 구성하는 프론트엔드 라이브러리다. 대시보드, 거래 목록, 필터 패널, drill-down 화면처럼 상태 변화가 많은 화면에 적합하다.

### TypeScript

프론트엔드 코드에 타입을 부여한다. 후잉 응답, 거래 모델, 집계 구조를 다룰 때 데이터 형태를 명확히 해주고 리팩터링 안정성을 높여준다.

### Vite

프론트엔드 개발 서버와 빌드 도구다. 설정이 가볍고 개발 속도가 빠르다. SEO가 중요하지 않은 개인용 SPA에 잘 맞는다.

### TanStack Query

API 데이터를 가져오고 캐시하는 계층이다. `이번 달 거래`, `카테고리 집계`, `계정 잔액` 같은 조회 데이터를 효율적으로 재사용하게 해준다.

### TanStack Table

거래 탐색 화면의 핵심 도구다. 정렬, 필터, 컬럼 표시, 페이지네이션 같은 기능을 유연하게 조합할 수 있다.

### Tailwind CSS

스타일링 도구다. 컴포넌트 단위로 빠르게 화면을 만들 수 있고, 레이아웃과 간격 체계를 일관되게 유지하기 쉽다.

### Recharts

월별 추이, 카테고리 비중, 계정별 변화 같은 시각화를 맡는다. React와 결합이 단순하고 초기 생산성이 좋다.

### Go

백엔드 API와 동기화 작업을 담당한다. 후잉 API 호출, 데이터 정규화, 집계 계산, 스케줄 작업을 처리한다.

### Chi

Go에서 HTTP API 라우팅을 구성하는 얇은 프레임워크다. 필요한 기능만 쓰면서 구조를 단순하게 유지할 수 있다.

### Postgres

후잉 원본을 대체하지 않는 로컬 캐시 저장소다. 거래 스냅샷, 월별 집계, 저장된 뷰, 사용자 설정을 보관한다.

### Nginx Proxy Manager

이미 개인 서버에서 서비스 라우팅에 쓰고 있는 리버스 프록시다. whoofolio는 별도 프록시를 내장하지 않고, 기존 운영 방식에 편승하는 편이 단순하다.

## 왜 Go인가, 왜 Node가 아닌가

### Go를 권하는 이유

- 단일 바이너리로 배포가 쉽다
- 메모리 사용량이 안정적이다
- 스케줄 작업과 HTTP API를 한 프로세스로 묶기 좋다
- 개인 서버에서 장기 운영 시 관리 피로가 낮다

### Node를 고르면 좋은 점

- 프론트와 언어를 통일할 수 있다
- 구현 속도가 빠를 수 있다
- 생태계가 넓다

### 이번 프로젝트에서 Go를 택하는 이유

이 프로젝트는 실시간 협업 서비스가 아니라, 개인 서버에서 오래 켜둘 조회 서비스다. 운영 단순성과 안정성이 중요하므로 Node의 개발 편의보다 Go의 배포/운영 단순성이 더 유리하다.

## 왜 Postgres인가, 왜 SQLite가 아닌가

### Postgres를 권하는 이유

- 필터와 집계가 늘어나도 버티기 쉽다
- 인덱스 설계와 쿼리 확장이 안정적이다
- Docker 백업/복구 패턴이 익숙하다
- 나중에 배치 집계나 materialized view 성격의 구조로 확장하기 좋다

### SQLite를 고르면 좋은 점

- 설정이 거의 없다
- 단일 파일이라 시작이 빠르다
- 매우 작은 개인 앱에는 충분할 수 있다

### 이번 프로젝트에서 Postgres를 택하는 이유

`whoofolio`는 단순 메모장이 아니라 거래 탐색과 월별 집계가 핵심이다. 읽기 성능과 운영 예측 가능성을 생각하면 초기부터 Postgres가 더 낫다.

## 왜 Next.js가 아닌가

- 공개 웹서비스가 아니므로 SEO 이점이 거의 없다
- SSR이 필수인 화면이 아니다
- 서버와 프론트를 한 프레임워크에 묶는 이점보다 분리된 SPA + API 구조의 단순성이 더 크다

## 왜 Tauri가 아닌가

- 목표 배포 환경이 개인 서버 + 브라우저이기 때문이다
- 데스크톱 앱 패키징은 현재 문제를 해결하지 않는다
- 운영과 배포 경로가 괜히 복잡해진다

## Wealthfolio와 왜 다른가

Wealthfolio는 본질적으로 로컬 퍼스트 투자 포트폴리오 앱이다. 데이터 저장, 도메인 모델, 앱 실행 방식이 그 목적에 맞게 설계돼 있다.

반면 `whoofolio`는 다음 점에서 다르다.

- 후잉이 원본 원장이다
- 핵심 가치는 쓰기가 아니라 조회다
- 실행 환경이 데스크톱이 아니라 개인 서버 웹앱이다
- 로컬 DB는 원장이 아니라 캐시와 집계 저장소다
- 기술 선택도 투자 앱의 로컬 엔진보다 단순 운영에 맞춰진다

즉, `whoofolio`는 Wealthfolio의 디자인 감각은 참고할 수 있어도, 구조 전체를 따라가면 안 된다.

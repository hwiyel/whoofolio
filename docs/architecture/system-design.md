# 시스템 설계

## 핵심 결정

`whoofolio`는 데스크톱 앱이 아니라 개인 서버에 올려두고 브라우저로 접속하는 작은 웹 서비스로 간다. 기본 실행 환경은 Docker이며, 후잉 데이터는 주기적으로 동기화해 로컬 캐시에 저장한다.

## 권장 아키텍처

```text
브라우저
  -> whoofolio 웹 앱
  -> whoofolio API
  -> 후잉 API
  -> 로컬 캐시 DB
```

## 구성 요소

### 프론트엔드

- React 기반 SPA
- 월간 대시보드, 거래 테이블, 필터, drill-down에 집중
- 개인용 서비스이므로 SSR은 필수가 아니다

### 백엔드 API

- 후잉 인증과 동기화 처리
- 후잉 응답을 UI 친화적인 모델로 정규화
- 최근 데이터와 집계 캐시 유지
- `/overview`, `/transactions`, `/categories` 같은 조회 중심 API 제공

### 로컬 데이터베이스

- 역할: 캐시, 파생 집계, 저장된 필터, 사용자 설정
- 회계 원장의 기준점으로 취급하지 않는다

## 데이터 전략

- 후잉에서 주기적으로 pull 하고, 필요 시 수동 새로고침을 지원한다
- 정규화된 거래 스냅샷과 월간 집계 테이블을 저장한다
- 월별 합계, 카테고리 통계, 계정 추이를 미리 계산해 둔다
- 동기화 문제 추적을 위해 원본 응답 참조값을 남길 수 있게 한다

## 기술 스택 결정

초기 구현 기준 권장 스택은 아래와 같다.

- 프론트엔드: React + TypeScript + Vite
- 상태/데이터: TanStack Query + TanStack Table
- 스타일: Tailwind CSS
- 차트: Recharts 또는 ECharts
- 백엔드: Go + Chi
- 데이터베이스: Postgres
- 리버스 프록시: 기존 Nginx Proxy Manager

## 왜 이 스택인가

### React + TypeScript + Vite

- 대시보드와 테이블 UI를 빠르게 만들기 좋다
- 생태계가 넓어서 필터링, 차트, 테이블 조합이 쉽다
- 개인 프로젝트에서 구현 속도가 빠르다

### TanStack Query + TanStack Table

- 조회 중심 제품에 잘 맞는다
- 필터, 정렬, 페이지네이션, 캐시 전략을 깔끔하게 가져가기 좋다
- drill-down 중심 인터랙션을 만들기 편하다

### Go + Chi

- Docker 배포에 유리하다
- 메모리 사용량이 가볍고 단일 바이너리 운영이 쉽다
- 주기 동기화 작업과 간단한 API를 한 프로세스에 묶기 좋다
- 개인 서버에서 Node보다 운영 피로도가 낮다

### Postgres

- 거래 필터링, 월별 집계, 인덱스 설계가 안정적이다
- Docker 백업/복구가 익숙하고 단순하다
- SQLite보다 운영이 조금 무겁지만, 조회와 집계가 많아질수록 이점이 크다

### Caddy

- HTTPS 설정이 단순하다
- 개인 서버에서 리버스 프록시 운영 난도가 낮다

## 대안과 판단

- `Next.js`: 가능하지만 이 프로젝트는 SEO가 중요하지 않고, 서버 액션이나 SSR 이점이 크지 않다. 지금은 Vite SPA가 더 단순하다.
- `Node.js API`: 구현 속도는 빠를 수 있지만, 개인 서버 장기 운영 관점에서는 Go 쪽이 더 단정하다.
- `SQLite`: 아주 작은 시작점에는 괜찮지만, 나중에 월간 집계와 필터가 늘어나면 Postgres가 더 편하다.
- `Tauri`: 개인 서버 배포와 맞지 않는다. 이 프로젝트에는 과하다.

## 핵심 트레이드오프

- SPA + API 분리는 단순하고 Docker 운영과 잘 맞는다
- 로컬 캐시는 복잡성을 늘리지만, 조회 속도와 API 호출 절약에 충분한 가치가 있다
- read-only 우선 전략은 위험을 줄이고 정보구조를 먼저 검증하게 해준다
